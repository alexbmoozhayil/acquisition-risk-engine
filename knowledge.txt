# Acquisition Risk Engine Knowledge

- This project is a Java companion to Acquisition Intel.

- Acquisition Intel focused on RAG, search, graph, and AI workflows.

- Acquisition Risk Engine focuses on Java, Spring Boot, SQL, and risk scoring.

- Spring Boot is the Java backend framework for this project.

- Spring Boot plays a similar role to FastAPI in the Python project.

- Maven manages Java dependencies and build commands.

- The Maven wrapper, mvnw, lets the project run Maven consistently.

- spring-boot-starter-web adds REST API support and an embedded web server.

- A controller receives HTTP requests and returns HTTP responses.

- @RestController marks a Java class as a REST API controller.

- @GetMapping("/health") maps GET /health to a Java method.

- Spring Boot can convert a Java Map into a JSON response.

- The /health endpoint verifies that the service is running.





- A database migration is a versioned SQL file that changes the database schema.

- Flyway runs migration files automatically when the Spring Boot app starts.

- V1__create_initial_tables.sql is the first migration for this project.

- Flyway created the agencies, vendors, and contracts tables.

- flyway_schema_history tracks which migrations already ran.

- Spring Boot needs spring-boot-starter-flyway for Flyway integration.

- The database schema is now managed through code instead of manual SQL steps.

- PostgreSQL is running in Docker as acquisition-risk-postgres.

- The app connects to PostgreSQL using JDBC through application.properties.


I added PostgreSQL as the structured data store and used Flyway to manage schema changes through versioned SQL migrations. Instead of manually creating tables, the database schema is defined in code and automatically applied when the Spring Boot app starts. This makes the backend easier to reproduce, test, and deploy.


- V2__seed_sample_contracts.sql inserts sample contract data.

- Flyway runs V2 after V1 because migrations run in version order.

- JdbcTemplate lets Java run SQL queries against PostgreSQL.

- A repository class handles database access.

- A controller class handles HTTP requests.

- ContractResponse is a Java record used for the API response shape.

- GET /contracts reads contract rows from PostgreSQL and returns JSON.

- SQL joins combine contracts with vendor and agency names.

- We used seed data before external API ingestion to test one layer at a time.


- GET /vendors returns vendor-level summary data.

- VendorResponse is the JSON response shape for vendors.

- VendorRepository runs SQL for vendor queries.

- LEFT JOIN keeps vendors even if they have no contracts.

- COUNT counts contracts per vendor.

- SUM adds total contract value per vendor.

- COALESCE replaces null with 0.

- GROUP BY creates one summary row per vendor.

- Vendor-level summaries prepare us for vendor risk scoring.



- GET /vendors/{vendorId}/contracts returns contracts for one vendor.

- @PathVariable reads values from the URL path.

- WHERE v.id = ? filters the SQL query by vendor ID.

- JdbcTemplate passes the vendor ID as a query parameter.

- Query parameters are safer than building SQL strings manually.

- Nested endpoints are useful when one resource belongs to another resource.

- Vendor contract history is needed before calculating vendor risk.
